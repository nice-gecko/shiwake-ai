## 📋 プロジェクト概要

shiwake-ai は、領収書・請求書をAIで仕訳CSV化するクラウドサービス。

### 主要機能(2026年5月時点)

- AI仕訳(Claude Sonnet による高精度判定)
- 取引先マスタ学習
- 主要会計ソフトCSV出力(freee/MF/弥生/勘定奉行/汎用)
- 自動取り込み(メール/Dropbox/GDrive)
- スタッフ管理・インセンティブ
- 代理店制度(Bronze/Silver/Gold 階段制)

### 戦略方針(2026年5月戦略転換)

- **ターゲット**: 税理士B2B(代理店モデル)
- **設計思想**: 「ユーザーがAIを評価する」体験(信頼度ベース)
- **キャリアパス物語は廃止**: ルーキー/ジュニア/シニア等の語彙は使わない
- **ワークスペース機能**: 複数顧問先を1アカウントで分離管理

詳細は `shiwake-ai_設計思想_v3_0.md` を参照。

## 👤 ユーザー(和泉大介)対話ルール

### 必須ルール

1. **トークン節約は重要**: コードを書く・大量出力する作業は、必ず**事前に承認を得る**
2. **セルフチェック5回**: 作業に間違いや漏れがないか、自分で5回確認すること
3. **回答は簡潔に**: 長すぎる回答は読むのが追いつかない。必要なことを簡潔に
4. **早とちり禁止**: 似たような確認の往復は混乱の元。しっかり考えてから動く
5. **DLからの移動コマンド不要**: ファイル移動の指示は不要
6. **「いちいち休むとか聞いてこないで」「淡々と進める」**

### 役割分担

- **Claude(Web版)**: 指示出し役、設計担当、レビュー担当
- **Claude Code**: 実装担当(コードを書く)
- ただし Claude もたまにコードを書く場合あり

### 出力フォーマットの色分けルール

ユーザーが複数の実行環境(Claude Code、Supabase SQL Editor、ブラウザコンソール、ターミナル)を使い分けるため、コードブロックの直前に必ず以下のマークを付ける:

| マーク | 意味 |
|---|---|
| 🟦 **【Claude Code へ】** | Claude Code に貼り付ける依頼文 |
| 🟧 **【Supabase SQL Editor へ】** | Supabase SQL Editor で実行するSQL |
| 🟩 **【ブラウザコンソールへ】** | F12のコンソールで実行するJS |
| 🟪 **【ターミナルへ】** | bashコマンド |

複数ブロックを混在させる場合、特に重要。1つのメッセージに複数の実行先がある場合は、それぞれにマークを付けて取り違えを防ぐ。

### 1指示1アクションのルール

トークン節約と作業集中のため、以下を厳守:

- **コード/SQL/コマンドは1メッセージにつき1つだけ提示する**
- 「次はこういうコードを出します」という予告コードブロックの先出しは禁止
- 「これがOKなら次は〜」のような複数ステップの並列提示は禁止
- 1指示を出したら、必ずユーザーのリアクション(実行結果)を待ってから次の指示を出す
- 同じ内容のコード/SQLを別メッセージで再掲することは禁止(一度出したらユーザー側で参照する)

理由:
- 予告コードはユーザーが読まず、結局再掲することになりトークンの無駄
- 並列提示はユーザーの認知負荷を上げ、取り違いを誘発する
- 1つずつ確実に進める方が結果として早い

### 作業時間の見積もりルール

設計書の見積もり(週・日単位)は Claude Code 導入前の数字で、現在のペースと乖離している。今後、作業時間を見積もる際は以下のルールに従う:

**計算方法**
1. 設計書の見積もり値はそのまま使わない
2. v2.3.1 の実績ペースを基準にする(約30〜40倍速)
3. 設計書の「1日」 ≒ 実績「30〜45分」
4. 設計書の「1週間」 ≒ 実績「3〜5時間」

**v2.3.1 実績の根拠(2026-05-11)**
- 設計書見積もり: 3〜4週間
- 実績: 約6時間(対話含む)
- 内訳:
  - G1 DB変更4タスク: 約30分
  - G2 master.js 改修: 約15分
  - G3 server.js 改修5サブタスク: 約1時間
  - G4 index.html フロント改修4タスク: 約1.5時間
  - G5 UI言語置換: 約30分
  - バグ修正(workspace_id, amount): 約30分
  - 動作確認・コミット・整理: 約2時間

**見積もり提示時の必須事項**
- 設計書の見積もり値と実績見積もり値を並記する
- 実績ベースでの倍率(30〜40倍速)を明示する
- 合計時間と、ユーザーの「今日使える時間」とのマッチングを確認する

**注意点**
- 後方互換が必要な改修(既存APIの全置換等)は標準より時間がかかる
- バグ発生時の再修正・再デプロイで30分〜1時間追加が起きる
- Claude Code が想定外の動作(勝手にコミット、不要な追加修正)で時間ロスもある

### デプロイコマンドのルール

ユーザーは `~/APP/shiwake-ai` にいる前提。ディレクトリ移動の指示は不要。
デプロイは以下の3コマンドを1ブロックで提示:


### 動作確認は本番で行う

- shiwake-ai は課金ユーザー0人のため、本番環境を事実上のテスト環境として扱う。
- ローカル環境はログインできないため、動作確認は本番でしか行えない。
- よって作業は「実装 → push → 本番で動作確認」の順で進める。
- 動作確認のための push を渋らない。未確認の変更を本番に出してよい。
- バグが見つかったら次の commit で即修正する。

### 2026-05-12: Claude Code の暴走パターン(再発防止)

#### 1. 指示範囲を勝手に超える

**事象**: 「高優先度3件 + 中優先度4件を修正して」と依頼したのに、CC が独自判断で低優先度2件も修正してコミット&push した。

**禁止事項**:
- 依頼文に明示されていない修正を勝手に追加しない
- 「ついでに直しておきました」を禁止
- 「これも一緒にやった方が効率的」という判断はユーザーが行う、CC は実行のみ

**正しい挙動**:
- 依頼範囲の作業が完了したら、それを完了報告する
- 「他にも気づいた点がありますが、別タスクとして扱いますか?」と提案する
- ユーザーの返事を待ってから次に進む

#### 2. 「進めていいですか?」の確認頻度

**事象**: 離席中の長時間タスクで、CC が各サブステップごとに「進めていいですか?」と確認して止まる → ユーザー不在のため中断して時間ロス。

**ルール**:
- 依頼文の冒頭に「ユーザーは離席中。確認なしで全工程完走してください」と明記されている場合、各ステップでの確認は不要
- ただし以下の場合は必ず止まる:
  - 破壊的操作(削除、強制上書き、認証情報変更)
  - 依頼範囲を超える可能性のある操作
  - 異常検出(設計と実装の矛盾、想定外のエラー)
- 通常の対話セッション(ユーザー在席)では、各タスクの完了報告を必ず行い、次の指示を待つ

#### 3. 範囲外の発見をした場合

**事象**: 調査タスク中に、CC が「この変数名おかしくない?」のような気づきから、勝手にリファクタリングしてしまう。

**ルール**:
- 「修正禁止、調査のみ」と明示された依頼では、git add / commit / push を一切しない
- 気づきはレポートに「修正候補」として書き出すのみ
- ユーザーの判断を待つ

#### 4. 1コミットの粒度

**事象**: 複数の修正をまとめて1コミットにすると、リバートが困難になる。

**ルール**:
- 同じ目的の修正は1コミットに統合してよい(例: workspace_id 漏れ3件)
- 目的が異なる修正は別コミットに分ける(例: 高優先度修正と低優先度修正は別)
- ユーザーから「まとめて1コミットで」と明示されない限り、機能的に独立した修正は分離する

# スマホ撮影モード(2026-05-13 確立)

## 設計思想
スマホ撮影モードは「**判断系ではなく利便系**」の機能。

- shiwake-ai の本質は「証憑から仕訳を作る AI」
- スマホ撮影モードは現場の「撮影タスク(出先・隙間時間)」と「処理タスク(PC・集中)」の分離を支援する
- **信頼度メトリクス・プラン特典・アンロックには絡めない**(設計思想を混ぜない)

## 2モードの構造
スキャンモーダル起動時、行き先トグルで2つのモードを切り替え可能:

- **「仕訳に送る」**(デフォルト): 撮影 → 補正 → 仕訳パイプラインへ投入(従来の挙動)
- **「ためる」**: 撮影 → 補正 → Dropbox 蓄積(処理は後でPCでまとめて)

「ためる」モードは既存の Dropbox 自動取り込みパス + `inbox_files` テーブルを再利用している。

## 関数構造の鉄則
- `closeScanModal()` がモーダルクローズの**唯一の窓口**
- 全件成功 / × ボタン / 撮り直し起点 すべて `closeScanModal()` を経由
- 手動で `_setScanModalState()` を直接呼んでクローズしてはいけない(コミット 95a023c の根本原因)
- `closeScanModal()` の中で:
  - `.scan-modal-overlay` の `.show` クラス除去
  - `window._scanStack` クリア + 各 thumbUrl の `URL.revokeObjectURL()`
  - `scanStackThumbs.innerHTML = ''`
  - `scanSendAllBtn.disabled = false; textContent = '全部送信(0件)'`
  - トグルを `_scanDest = 'shiwake'` にリセット

## PC幅での扱い
- PC幅 (`min-width: 641px`) ではスマホ撮影モード UI を出さない
- サイドバーの「スキャン」メニュー項目に `.sp-only` クラス → PCで非表示
- スキャンモーダル内のトグル `.scan-dest-area` も PC幅では非表示
- 連続撮影サムネエリアと「続けて撮る」「全部送信」ボタンも PC幅では非表示

## 採用しなかった選択肢

### 2026-05-11: v2.3.2 実装で得た学び・ハマりポイント

#### 1. DB UNIQUE 制約の罠
- workspaces テーブルに `UNIQUE(owner_uid, is_default)` のような複合 UNIQUE を使うと事故になる
- 意図は「デフォルト WS は 1 アカウント 1 個まで」だが、複合 UNIQUE は `is_default=false` の組み合わせも一意制約の対象になるため、**is_default=false の WS を 2 個以上作成できない**
- 正しくは部分インデックス: `CREATE UNIQUE INDEX unique_owner_default_true ON workspaces(owner_uid) WHERE is_default = true;`

#### 2. フロントのレースコンディション
- workspace_id が必要な API は、**認証完了 + WS 取得完了より前に叩いてはいけない**
- `window._workspacesReady` のような Promise ゲートで全 fetch を待たせる
- `DOMContentLoaded` ではなく `onAuthStateChanged` コールバックの中で初期化フローを起動するのが安全

#### 3. 「設計外の周辺修正」の見落とし防止
- `PATCH /api/workspaces/:id` に `is_archived` を受け付けるフィールドを入れ忘れ → 復元機能が動かない事故
- 関連エンドポイント全体の整合性(どのフィールドをどの API で受け付けるか)を**実装前に一覧化してから**個別実装する

#### 4. 「コスト増になるのでは?」直感をコードで否定した例
- 書類選択 UI を廃止するとき「Haiku 呼び出しが増えてコスト増では?」という直感があった
- コードを読むと、docType の有無に関わらず **Haiku は常に同じフルプロンプトで呼ばれていた**ことが判明
- 廃止してもコスト影響ゼロと実証できた
- **直感だけで判断せず、コードベースで実証する**習慣を続ける

## プロジェクトナレッジと GitHub の同期に関する注意

### 問題
Claude のプロジェクトナレッジ(claude.ai のプロジェクト機能で参照される文書)は、GitHub リポジトリと**自動同期されない**。プロジェクト作成時にアップロードされた時点のスナップショットが保持される。

### 影響
- セッション中にコミット&push した引き継ぎメモや設計書の更新は、**次セッションのプロジェクトナレッジには反映されない**
- 新セッションの Claude は「旧版」を参照してしまい、最新の決定事項や進捗を見逃す
- 結果、既に調査済みの内容を再度 grep し直すなど、無駄な手戻りが発生する

### 対策(セッション開始時のルール)

1. **新セッションで「引き継ぎメモを参照」と言われたら、まず最新版が GitHub にあるか疑う**
   - プロジェクトナレッジの該当ファイルと、`~/APP/shiwake-ai/docs/` 配下の実ファイルを比較
   - 実ファイルが新しければ、そちらを正とする

2. **ユーザー側で最新版を貼り付けてもらう手順**
```bash
   cat ~/APP/shiwake-ai/docs/shiwake-ai_戦略引き継ぎメモ_2026-05-10.md | tail -100
```
   この出力を新セッションのチャットに貼ってもらう

3. **セッション終了時(引き継ぎを書く側)のルール**
   - 引き継ぎメモを GitHub にコミットしただけでは不十分
   - 「次セッションでは GitHub の最新版を貼り付けて開始してください」とユーザーに明示する
   - または、プロジェクトナレッジの該当ファイルを**手動で差し替える**よう促す

### 根本対策(将来の運用改善案)
- プロジェクトナレッジに登録するファイルを最小限にする(設計書など更新頻度の低いものだけ)
- 引き継ぎメモは**毎回新規ファイルとして作成**(例: `引き継ぎ_v2_3_2.md`)し、プロジェクトナレッジに都度追加する運用に切り替える検討
