## 1. 思想転換の背景

### 1.1 兄(税理士)からのフィードバック

> **「第一に信頼性。間違えると税務調査リスクで、自分の信用が傷つく」**

税理士業界の本音。仕訳ミスは:
- 顧問先の決算・申告に直接影響
- 税務調査での否認リスク
- 税理士自身のレピュテーション毀損

つまり「**精度の高さが何より重要**」であり、「**成長物語**」のような演出は業務文脈ではノイズになる。

### 1.2 ユーザー(=代表)自身の違和感

開発の中で、5階層キャリアパス(ルーキー〜エリート)の演出が「**やや幼稚**」と感じる場面が増えた。

- BtoB業務ツールに対して、ユーザーは**プロらしさを期待**する
- 「卒業」「昇格」のメタファーは、プロの自尊心と噛み合わない
- 税理士向けの代理店制度には「成長物語」が機能するが、エンドユーザーには逆効果

### 1.3 「信頼性」を体験化する設計の必要性

兄の「**50件くらい慣らし運転で精度を見たい**」というFBに、設計の本質がある。

これは:
- 「AIに自分の業務を任せていいかを、自分の目で確認したい」
- 「数字で精度を確認できれば、自分の判断で自動化を進められる」
- 「AIに勝手に進められるのではなく、自分が決めたい」

という**プロの主体性の表れ**。これを設計に反映させる必要がある。

## 2. 新しい設計思想の核

### 2.1 一文で表すと

> **AIを評価するのはユーザー。AIは数字で自分を見せる。**

### 2.2 三つの原則

#### 原則1: 信頼度の数字で語る

「育成中⭐⭐⭐⭐⭐」のような曖昧な表現を排除し、すべて**数字**で見せる:

- **承認率**: AIの提案がそのまま承認された割合(例: 152件中147件 = 96.7%)
- **修正パターン**: どの項目が最も修正されているか(例: 勘定科目3件、税区分1件)
- **学習済み取引先**: AIが認識できる取引先数(例: 23社)
- **マスタヒット率**: 過去の学習が効いている割合

これらが**常時ダッシュボードに表示**され、ユーザーはいつでも「AIが今どれくらい使えるか」を確認できる。

#### 原則2: ユーザーが自分で開放する

旧設計では「件数達成→AI判定→アンロック」という**AIがユーザーを評価する構造**だった。

新設計は逆転する:

```
[ユーザーがAIを試用]
       ↓
[精度の数字を見て判断]
       ↓
[「もう信頼できる」と思ったら自分で機能をON]
```

4つの自動化機能(取り込み・エクスポート・ルール学習・承認)は、**最初から全部表示**される。ユーザーは数字を見て、好きな順番・好きなタイミングでONにできる。

これは「**プロの判断を尊重する**」設計。

#### 原則3: 透明性が継続利用を生む

旧設計から継承する核心:

> **「何をどう判断させるかを明確に見せること」が継続利用につながる**

これは v2.0.0 から変わらない。新設計ではこれをさらに強化する:

- AIが何を見て判断したか(ソース表示)
- AIが何を学習したか(ルール一覧)
- AIが何を間違えたか(修正履歴)
- AIが何が苦手か(修正パターン分析)

すべて見える化することで、ユーザーは「**自分専用ツールを育てている**」感覚を維持できる。

## 4. 信頼度メトリクス(新規実装が必要)

### 4.1 記録すべきデータ

仕訳イベントごとに以下を記録する必要がある:

```sql
-- shiwake_records テーブル(既存)に列追加
ALTER TABLE shiwake_records
  ADD COLUMN ai_proposed_debit text,
  ADD COLUMN ai_proposed_credit text,
  ADD COLUMN ai_proposed_tax text,
  ADD COLUMN ai_proposed_memo text,
  ADD COLUMN was_modified boolean,
  ADD COLUMN modified_fields text[];  -- ['debit', 'tax'] など
```

これにより以下が計算可能になる:

- **承認率**: `was_modified = false の件数 / 全承認件数`
- **修正パターン**: `modified_fields の集計`(どの項目が最も修正されたか)
- **項目別精度**: 勘定科目精度、税区分精度、メモ精度をそれぞれ計算

### 4.2 表示するメトリクス

| メトリクス | 計算式 | 表示例 |
|---|---|---|
| **承認率** | 修正なし承認件数 / 全承認件数 | 96.7% (147/152件) |
| **項目別精度** | 項目ごとの修正なし率 | 勘定科目 98%, 税区分 95% |
| **修正傾向** | 修正された項目の頻度順 | 勘定科目(3件) > 税区分(1件) |
| **学習済み取引先** | マスタ登録済み取引先数 | 23社 |
| **マスタヒット率** | マスタヒット件数 / 全件 | 78% |
| **異常検知件数** | 統計的外れ値検出件数(将来) | 0件 |

### 4.3 ダッシュボードの設計方針

- **常時表示**: メイン画面に常に精度サマリーを表示
- **詳細展開**: クリックで詳細画面へ(項目別精度、修正履歴、傾向グラフ)
- **時系列**: 「直近30日」「全期間」で切替可能
- **取引先別**: 「この取引先のAI精度はどれくらいか」を確認可能

## 5. 4自動化機能の再定義(機能はそのまま、開放方法のみ変更)

### 5.1 機能仕様(変更なし)

旧設計で定義された4機能の**機能仕様自体は変更しない**:

| 機能 | 役割 | 実装バージョン |
|---|---|---|
| **自動取り込み** | メール/Dropbox/GDriveから自動取得 | v2.3.0(実装済) |
| **自動エクスポート** | 会計ソフト別CSV自動整理 | v2.4.0(未実装) |
| **自動ルール学習** | 繰り返しパターンの自動ルール化 | v2.5.0(未実装) |
| **自動承認** | 高信頼度仕訳の自動確定 | v2.6.0(未実装) |

### 5.2 開放方法(変更あり)

旧設計:
```
件数達成 → AI判定 → アンロック → 機能ON
```

新設計:
```
ユーザーがダッシュボードで信頼度確認 → 自分の判断で機能をON
```

**すべての機能は最初からUI上に表示される**。ただし:
- 各機能を有効化するには**プラン要件**を満たす必要がある(Agent版以上等)
- プラン要件を満たしていない場合は「Agent版にアップグレード」CTAを表示
- **Agent版以上のユーザーは初日から全機能を有効化可能**

### 5.3 推奨利用シナリオ(任意のガイダンス)

機能の有効化を強制はしないが、ガイダンスとして:

> 💡 **おすすめの試し方**
> 1. まずは50件ほど手動で承認し、精度を確認
> 2. 承認率が90%を超えたら「自動取り込み」をON
> 3. さらに使い込んで95%超えたら「自動承認」もON

これは**強制ではなく提案**。ユーザーは無視してすぐ全部ONにすることもできる。

### 5.4 ロールバックの保証

任意のタイミングで自動化機能をOFFにできる。一度ONにしたら戻れない、という制約は設けない。

## 7. 維持される設計(変更しない要素)

### 7.1 代理店制度(Bronze/Silver/Gold)

代理店向けの育成物語は**維持**:

- 代理店は「稼ぎたい」モチベーションで動く
- ゲーミフィケーションが効く
- 階段制マージンは既に v2.2.0 で実装済み
- 兄(税理士)を含む税理士事務所が代理店として参加する構造

代理店ダッシュボードのランク表示、Bronze→Silver→Goldの昇格演出はそのまま。

### 7.2 4自動化機能の機能仕様

機能の中身は変えない。開放方法だけが変わる。

### 7.3 業法ガード

shiwake-ai は「CSV出力する計算補助ツール」という法的位置づけを維持。
自動承認機能を持つが、最終的な確定行為は会計ソフトで人間が行う構造は崩さない。

### 7.4 5プラン構造(AI SaaS / Agent / Elite)

価格・含件数・機能差分は v2.2.0 の設計を維持:

| プラン区分 | 主な顧客 | キャリアパス連動 |
|---|---|---|
| AI SaaS版 | 個人事業主、小規模法人 | 連動なし(自動化機能制限) |
| Agent版 | 中小法人、税理士顧問先 | 連動なし(全自動化機能利用可) |
| Elite | 大規模法人、税理士事務所主要顧問先 | 連動なし(対話モード等の差別化機能) |

**プラン名から「ルーキー」「エリート」のキャリアパス意味を切り離し、機能差別化の名前として使う**。Eliteは「最上位プラン」「対話モード搭載プラン」として位置付け、キャリアパスの最終形態という意味は捨てる。

### 7.5 業種テンプレート・Myテンプレート

機能仕様は変更なし(v2.8.0、v2.9.0で実装予定)。

### 7.6 異常検知アラート(常時機能)

「自動化が進むほど必要な安全装置」という位置づけは維持。
信頼度ベース設計においてもこれは重要(ユーザーが自動承認をONにした際の安心材料として機能)。

## 9. Eliteプランの位置づけ(再定義)

### 9.1 Eliteの本質的価値

旧設計では「5階層キャリアパスの最終形態」だったが、新設計では:

> **対話モード+文脈横断判断を持つ、もう一人の経理担当者**

として再定義する。

「成長の到達点」ではなく、「**最初から対話できる経理AI**」というポジショニング。

### 9.2 Eliteで実現する体験

Slack/Teams/LINE/音声で:

```
経理担当者: 「先月の懇親会の領収書、これ会議費でいい?」
Elite     : 「過去の同じ取引先パターンから、5万円以下なら会議費、
            5万円超えなら交際費でした。今回は3万円なので会議費で
            提案します。」
```

これは Agent版にはない「**会話による業務委任**」体験。

### 9.3 兄に見せる時の説明

Phase 4(自動承認)まで揃ったAgent版を見せた後:

> 「ここまでがAgent版です。Elite版はこれにSlack対話と文脈横断判断が加わり、月¥250,000です。実装は来年予定で、現在は構想段階。代理店としての商品ラインナップとしてイメージしてください。」

正直に伝える。Phase 4まで動く実物があれば、Eliteの構想にも説得力が出る。
