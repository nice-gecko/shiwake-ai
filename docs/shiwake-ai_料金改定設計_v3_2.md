## 1. 改定の背景と狙い

1. **競合比較は「入口の安さ」で決まる。** STREAMED基本料 ¥8,950 に対し、入口 ¥2,980 で明確に下回る。買い手は最初に見た数字で選ぶ。
2. **件数 = Claude APIコストに直結する真のコスト要因。** 課金を件数に寄せる＝筋の通った「使うほど増える」。
3. **WSはデータの仕切りでコストほぼ0。** 旧設計はWSを ¥20,000 の課金ゲートにしていた（コスト構造と逆）。WSは全パック無制限へ。
4. **旧「AI SaaS版／Agent版／Elite」3軸＋24プランは複雑。** 単一ライン4パックに統合し、構造をシンプルにする。

**モデルの骨格**: 低い入口（件数で受動的に育つ）＋ 能動的にパックを上げる。隠れ課金はしない —— 核思想「判断を見せる」と矛盾するため、課金見込みは管理画面で常時可視化する（§5）。

### 2.2 件数従量（込み件数を超えた分に課金）

各パックの「含む件数」（§2.1: basic 100 / automation 1,000 / complete 2,500 / dialog 無制限）を超えた分に、以下の逓減単価で課金する。下表の区分は「込み件数を超えてからの累積件数」に対する単価。

| 込み件数超過後の区分 | 単価 |
|---|---|
| 込み件数まで | パック料金に込み（¥0） |
| 超過 1〜400件 | ¥18/件 |
| 超過 401〜1,900件 | ¥16/件 |
| 超過 1,901件〜 | ¥15/件 |

- 逓減方式（区分ごとに単価が下がる累進）。たくさん使う事務所ほど超過分の実効単価が下がる＝育成思想と整合。
- `dialog` は含む件数が無制限のため、超過課金は発生しない。
- 旧モデルの「プラン別 included 件数 + 一律 ¥20/件」は廃止。逓減方式に統一。
- 単価設定の根拠: 業界ベンチマークはSTREAMEDの¥20/件（税抜・人力オペレーター込み）。shiwake-aiはAI完結でそのコストがないため全段¥20を下回る。下限を当初案¥12→¥15に引き上げたのは、APIコスト実測ログが未取得のため安全側に倒す判断（高解像度・複数枚証憑でも粗利を確保）。実測後に下げる余地は残す。
- DB実装方針: 逓減テーブル（`OVERAGE_TIERS`）は「設定値」として保持。込み件数はプランごとに異なるため、`calculateOverage` は込み件数を引数で受け取り、超過分にのみ逓減テーブルを適用する（P3-b2で改修）。

### 4.2 4パックの代理店卸売価格（Bronze 70%基準）

| パック | 通常価格 | Bronze卸売 | reseller plan_key |
|---|---|---|---|
| ベーシック | ¥2,980 | ¥2,086 | `reseller_basic` |
| オートメーション | ¥14,800 | ¥10,360 | `reseller_automation` |
| コンプリート | ¥24,800 | ¥17,360 | `reseller_complete` |
| ダイアログ | ¥98,000 | ¥68,600 | `reseller_dialog` |

- Silver/Gold は既存の Stripe Coupon（`silver_discount` / `gold_discount`）をそのまま適用。
- 取引高判定は従来どおり Bronze価格基準で集計。
- **クーポン確認済み（2026-05-16）**: `silver_discount`（percent_off 7.14%）/ `gold_discount`（percent_off 14.28%）は商品限定なしの率割引型のため、新4代理店商品にそのまま適用可能。新規作成・修正は不要。割引率もマージン階段制と整合（Bronze70%基準 → Silver 1−65/70=7.14%、Gold 1−60/70=14.28%）。P3のcheckoutで代理店ランクに応じ適用する。
- **件数従量分の代理店マージン適用は §7-3 で確定**（逓減従量に階段制をどう乗せるか未定）。

### 4.3 代理店規約の改定

**実物確認の結果、代理店規約 v1.0 は改定不要（2026-05-16）。** 規約は当初から「代理店プランの価格は別途料金プランページに記載」とし、プラン名・個別価格を本文に書かない設計。第5条のマージン表（Bronze 70%/Silver 65%/Gold 60%、取引高基準）は新体系でも有効。本文を全文スキャンした結果、旧プラン名・旧価格の記載はなし。マージン率・取引高判定の条文も変更不要。**法務監修は別途必須**（規約全体の最終確認として）。

### 4.4 料金ページの代理店料金トグル

料金ページにトグルを置き、通常価格 ⇔ 代理店価格（Bronze 70%基準）を切り替え表示できるようにする（P4-5で実装）。

- 表示範囲: 当初設計は「代理店価格は料金ページに出さない」だったが、誰でもトグルで切り替えられる方式に変更（2026-05-16決定）。卸売価格・マージン構造が一般顧客にも見える状態になることを承知の上での判断。
- トグルONで4カードの月額が代理店価格（reseller_basic ¥2,086 / reseller_automation ¥10,360 / reseller_complete ¥17,360 / reseller_dialog ¥68,600）に切り替わる。
- カードクリック時の `startCheckout` も、トグル状態に応じて `reseller_*` の plan_key を渡す。

## 9. スタッフインセンティブ機能（方針記録）

> 本セクションは方針の記録。機能の詳細設計（アクション定義・ギフト券配布の仕組み・閾値設定UI）は料金改定の完了後、独立タスクとして実施する。料金改定P1〜P6のスコープには含めない。

### 9.1 位置づけの再定義 — 「解約防止のコア施策」

当初は旧AI SaaS版限定のオプション機能だったが、本改定でコア機能に内包する。理由は、これが「あれば嬉しい機能」ではなく **継続率（事業の生死）に直結する施策** だと判明したため。

税理士事務所SaaSの最大の死因は、**契約者（所長）と使い手（スタッフ）が別人**であること。所長が導入を決める → スタッフ（多くは40〜50代の女性ベテラン勢）が新技術を敬遠して使わない → 形骸化 → 解約。インセンティブはこの「使い手が動かない」急所を直接突く施策である。

### 9.2 確定した設計方針

| 項目 | 内容 |
|---|---|
| 提供形態 | 全パックにコア内包（オプション課金にしない） |
| 有効化 | デフォルト無効。**オーナーID（税理士）の裁量で解放**する設定 |
| 原資 | 事務所持ち（shiwake-aiは集計とトリガーのみ担当。給与・労務には関与しない） |
| 報酬の形 | 実利ベース。例: 一定アクション到達でAmazonギフト券相当をスタッフIDのメールに配布（当初構想: 1,000件処理で500円分） |
| 指標 | **「修正・確認アクション数」ベース**（§9.3） |

ゲーミフィケーション（バッジ・ランキング等）単体では不十分という判断。ターゲット層（ベテランスタッフ）は実利が伴わない仕掛けを「子供騙し」と受け取るため、実報酬を軸にする。

### 9.3 指標を「件数」から「修正・確認アクション」へ

**課題**: 当初のインセンティブはAI SaaS版前提で「処理件数」を指標にしていた。SaaS版では所員が1件ずつ手を動かす＝件数が労力の代理指標として成立した。しかしAgent（自動取込・自動仕訳・自動エクスポート）では自動化が進むほど所員の労力＝件数が減り、所員は「画像を用意する人」になる。件数を指標に残すと「自動化で楽になった人ほど多く貰う」逆転や、「使うほど自分の取り分が減る」逆インセンティブが起き、最も使ってほしい層を遠ざける。

**解決**: 報酬対象を「処理した件数」ではなく **「AIを賢くした貢献＝修正・確認アクション」** に移す。Agentでも所員の仕事は消えていない（証憑を整える・AI提案を確認する・誤りを修正する）。この**修正こそがルール学習の燃料**であり、shiwake-aiの核思想「ユーザーが自分専用ツールを育てる」の"育てる手"を実際に動かしているのは所員である。

所員から見た物語: 「私が直す → AIが賢くなる → 賢くなった分が目に見える → 事務所からギフト券が届く」。手の動きと報酬が直結し、子供騙しにならない。所長にとっても、所員が修正を頑張るほどAI精度が上がり事務所が得をする＝原資に対して黒字の構造。

指標は **A案＝修正・確認アクション数**（「承認した」「修正した」回数）で確定。説明不要で納得感が高く、SaaSの件数モデルからの移行もわかりやすい。「学習貢献度」（修正の実際のルール化度合い）は理想的だが計測の透明性確保が難しいため、将来の発展形として保留。

### 9.4 「インセンティブの自然な減衰」を設計として肯定する

運用が進むとAIが賢くなり、修正アクションが減り、スタッフが受け取るギフト券も減っていく。これを **設計上の正しい挙動として肯定する**。

- 導入初期: 修正が多い → ギフト券がよく届く → スタッフの利用動機になる（**着火剤**）
- 運用が進む: AIが賢くなる → 修正が減る → ギフト券が減る。だがその頃には「AIの提案はそのまま通してよい」という体験と習慣がスタッフに定着している
- スタッフの心理: 「だんだん貰えなくなった」＝「なんだ、AI合ってるじゃん」という納得（賢くなったAIへの"憎らしさ"含む）。これ自体がAIへの信頼が育った証
- 所長の心理: 原資の支払いが減る＝AIが育った成果。支払い減少に不満が出ない構造

→ インセンティブは恒久的な報酬制度ではなく **「利用の着火剤」**。役目を終えたら自然に小さくなる。この減衰カーブを欠陥ではなく狙いとして設計・説明する。

### 9.5 今後のタスク（料金改定とは別）

- 「修正・確認アクション」の厳密な定義（何をカウントするか／重複や不正の防止）
- ギフト券配布の技術的な仕組み（配布API・原資管理・上限設定）
- オーナー向けの有効化／閾値設定UI（解放はオーナー裁量という前提のUI）
- 「判断を見せる」思想に沿った、スタッフ向けの貢献可視化画面

### 9.6 チート耐性（カウント不正対策）

**論点**: 報酬を「修正・登録の回数」に単純に紐づけると、スタッフが回数稼ぎに最適化できてしまう。同じ対象を直して戻してを繰り返す／無意味な微修正を連打する等。放置するとインセンティブ制度が崩壊し、かつAIに無意味なノイズを食わせて精度をむしろ下げる（制度の目的と正反対の結果）。

**原則**: カウント対象は「修正というアクション」ではなく「AIの学習に実際に効いた、新しく価値ある修正」のみ。

**採用方針（現時点の所感）**: 「同一対象の重複を弾く」を軸にする。理由＝マスタ登録はAPIコスト削減に直結する価値ある行為であり、新規登録はカウントする価値がある。一方、既に登録済みの内容と同じものを再登録・再修正してもAIにとっての価値はゼロなのでカウントしない。

実装イメージ:
- 登録・修正のログを参照し、対象が過去に保存済みの内容と同一かを判定する。
- チートに当たる操作（既存と同内容の再登録など）の際は、UI上で「これは以前に保存された内容です」「これに変更しますか？」のような確認を出し、その操作はカウント対象外とする。
- 「AIの提案を変えたか（＝AIが間違えていたか）」を基準にする案も思想的には理想だが、AIの提案の正誤を機械的に判定するのは難度が高い。まずは実装しやすい「同一対象の重複排除」から入る。

**残る穴と二段目の対策**: 重複排除は「同じものを繰り返す」チートには有効だが、「毎回ちがう無価値なものを作る」チート（架空の取引先を捏造して新規登録、実在取引先を細切れに別エントリ登録など）は止められない。捏造マスタはAIに汚いデータを食わせAIの精度をむしろ下げる。対策:

- **主軸（捏造の入口を塞ぐ）**: マスタ新規登録は「実際にAIが読み取った証憑から生成されたもの」に限定する。証憑と無関係な自由入力での新規登録を原則認めない。事後に無効化するのではなく入力段階で捏造を防ぐため、すり抜け競争が起きない。マスタが証憑に裏打ちされて育つのはチート対策以前に品質設計として正しい。
- **補強（事後の価値確定）**: 登録したマスタが後の仕訳で実際にヒットした時点でカウントする案。捏造マスタはヒットしないため報酬にならない。ただし登録〜ヒットのラグ、実在取引先の細切れ登録は両方ヒットしうる点が弱点。
- **例外運用**: AIが読み取れなかった取引先（手書き伝票、AI認識ミス）の手動追加は業務上必要なので可能とする。ただし手動追加分はカウント対象外（報酬にはならない）とし、捏造の穴を塞ぐ。
- **最後の歯止め**: カウントの内訳をオーナー（原資を出す税理士）が確認できる可視化。極端な異常検知（1日に多数の新規登録等）と組み合わせ、人の目を最終チェックとする。

詳細設計はインセンティブ機能本体の着手時に行う（料金改定P1〜P6のスコープ外）。
