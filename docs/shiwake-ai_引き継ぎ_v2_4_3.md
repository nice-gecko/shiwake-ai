### v2.4.2 緊急 race condition バグ修正(コミット 8fa7808)
- **症状**: 連続承認で未エクスポート仕訳が「黒穴に落ちる」(CSVに含まれず、`exported_at`だけセットされて再エクスポート対象から永久に消える)
- **真因**: `executeAutoExport()` 末尾の `UPDATE WHERE exported_at IS NULL` が SELECT 時点では存在しなかった行も巻き込んで更新
- **修正**: SELECT 直後に `recordIds = records.map(r => r.id)` を確定 → UPDATE を `id IN (recordIds)` 限定に
- merged モード分岐も不要になりシンプル化
- 復旧 SQL 実行済み(`5cb36889-...` の `exported_at = NULL`)
- 動作確認: 3件連続承認 → 3通のメール独立配送、全件処理(race解消確認)

### プラン上限バグ修正(コミット 04799ac4) ※動作確認は次チャット
- **発見経緯**: 「ライト 今月 144件（無制限）」表示の原因調査で発覚
- **真因**: server.js は `usage.limit` で返してるのに、index.html は `usage.monthly_limit` を読んでいた → 常に `undefined → null` → 全プラン無制限表示

- **影響範囲**: 全ユーザー・全プランで上限機能が事実上機能していなかった(課金ロジックの肝)

```
# v2.4.x UI改善 — 承認時のカウントアップアニメーションを画面右上のフローティングピルで追従表示

## 背景
承認時に「累計処理数」「取引先マスタ」の数字がカウントアップするアニメーションは既に実装されているが、対象カード(スタッフインセンティブ・取引先マスタ)が画面上部にあるため、ユーザーがスクロールして仕訳作業中だとアニメーションが見えない。

→ 元カードが画面外に出ているときだけ、右上にフローティングピルを出して、そこでカウントアップを再生する。

## 仕様

### A. 表示位置とスタイル
- 画面の **右上に固定(position: fixed; top: ~12px; right: ~12px)**
- フローティングピル(角丸 999px)
- 背景: `var(--color-background-primary)`(白)
- ボーダー: `0.5px solid var(--color-border-secondary)`
- 軽いシャドウ: `box-shadow: 0 2px 8px rgba(0,0,0,0.08)`
- パディング: `8px 14px`
- z-index: 既存のモーダルより低く、通常UIより高い値(z-index: 50 程度)

### B. ピル内のコンテンツ
2つの数値を横並びで表示(間に細い区切り線):

```
[累計 146 +1]  |  [マスタ 2 +1]
```

- 「累計」「マスタ」の小ラベル: 10px、`var(--color-text-secondary)`
- 数字本体: **14px、weight 500、青系の色 `#185FA5`**(v2.4.2 で使った色と同じ)
- 「+1」バッジ: 9px、`background: var(--color-background-success)`、`color: var(--color-text-success)`、`padding: 1px 5px`、`border-radius: 3px`
- 区切り線: `width: 0.5px; height: 14px; background: var(--color-border-tertiary)`

### C. 累計のリセットボタン
- 「累計」の右端に小さなリセットアイコンボタン(↻ や 🔄 等、既存のアイコンライブラリのスタイルに合わせる)
- クリック → 確認ダイアログ「累計処理数をリセットしますか?(取引先マスタはリセットされません)」
- 確認後、サーバに POST(エンドポイント要新規)
- リセット成功 → 累計表示が 0 にカウントダウンアニメーションで戻る

### C-1. リセット API(新規)
- `POST /api/user/reset-count`
- 認証必須(uid 認証ヘッダー)
- 処理: `users.monthly_count = 0` に更新(注意: monthly_count はサーバ側で今月の累積カウントを保持しているカラム)
- レスポンス: `{ success: true, monthly_count: 0 }`

### D. 表示制御(IntersectionObserver)
- 元カード(スタッフインセンティブカード or 取引先マスタカード)が画面内にあるとき: **フローティングピルを非表示**
- 元カードが画面外に出たとき: **フローティングピルを表示**
- 表示・非表示の切り替えは fade in/out アニメーション(150ms 程度)
- 両方のカードのうちどちらかが見えていれば非表示でOK(両方とも画面外のときだけピル表示)

### E. カウントアップアニメーション
- 既存の対象カード内アニメーションと**同じ実装ロジックを使い回す**(関数を共通化)
- 承認イベントが発火したときに:
  1. 元カードのカウントアップが走る(既存挙動)
  2. 同時にフローティングピルが画面外なら(=見えてれば)そちらでも走る
- アニメーション中は「+1」バッジを 1.5秒間 表示してフェードアウト(既存ロジックに合わせる)

### F. ユーザータイプ別の数字選択
- **インセンティブ対象の人**: ピルの左側は「インセンティブ対象数」を表示(既存の対象カードと同じロジック)
- **それ以外の人**: ピルの左側は「累計処理数」を表示
- 判定は既存実装(対象カードの表示判定)と同じヘルパーを使う

### G. レスポンシブ
- 画面幅が狭い(モバイル等)場合: ピルを少しコンパクトに(パディング縮小、ラベル省略可)
- 既存のレスポンシブパターンに合わせる

## 注意事項

### 必須遵守
- **CC push 忘れ再発防止**: 完了報告に必ず `git log origin/main -1` 出力を含めること
- 既存の対象カード(スタッフインセンティブ/取引先マスタ)内のカウントアップは**そのまま残す**(両方で同じアニメーションが走る)
- 既存の関数・ID・データフローは変更しない(IntersectionObserver と新規ピル要素を追加するだけ)
- リセット API は既存の認証パターンに合わせる
- 完了の定義は push origin main まで
- 5回セルフチェック

### 技術的注意
- position: fixed の z-index 衝突注意(モーダルが開いてる時はピルを隠す or 下層に)
- 既存のチャットボタン(画面右下)とは位置が違うので衝突しない
- スクロールイベントではなく IntersectionObserver を使う(パフォーマンス上有利)
- リセット処理は冪等性を持たせる(連打されても2回減らない)
```

## 4. 【重要】新運用方針 — 並行CC依頼の積極活用

### 4-1. 従来の運用
- 「先回り禁止のルール」を厳守: 1ターン1依頼、現タスク完了報告 → 次タスク依頼の順
- 理由: 並行で出されるとトークン無駄、判定と次着手がズレて混乱

### 4-2. 変更後の運用(2026-05-13 確定)
**独立して動かせるCC案件は、複数タブで並行投入OK**

#### 条件
- 各依頼が**他依頼の結果に依存しない独立タスク**であること
- 例:
  - ✗ **NG**: 「弥生分割設定UI改善」+「設定セクション再構成」(同じUIエリアに触るので順次)
  - ✓ **OK**: 「プラン上限バグ修正」+「カウントアップ追従バッジ」(別領域、独立)

#### 投入時のルール
- 依頼を投げるときに **「並行進行中: N個」** を必ず明示
- 完了報告は順次到着、順次処理する
- DSK が「混乱する」と感じたら 1個ずつに戻す

#### 既存ルールと矛盾しない範囲
- 1ターン内で複数依頼文を**書き出す**ことはOK(独立なら)
- 1ターン内で「現タスクの判定 + 同じ系統の次タスク依頼」を出すことはNG(従来通り)
- 例: バグ修正の動作確認待ち中に、別系統(UI改善等)の依頼を並行投入はOK

### 4-3. 期待効果
- セッション全体の時間短縮
- CC のアイドル時間を減らす
- DSK の待ち時間を減らす
