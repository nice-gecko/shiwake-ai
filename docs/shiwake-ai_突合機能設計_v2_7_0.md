## 1. 概要

### 1.1 目的

仕訳データと外部データ（銀行明細・カード明細・請求書）を照合し、差分・重複・欠落を自動検知する機能。税理士事務所の月次突合作業を大幅に効率化する。

### 1.2 設計思想との接続

- Agent = 「集める・整える・**気づく**」→ 突合は「気づく」の中核機能
- 取引先マスタにエイリアスが蓄積される → 「自分専用ツールを育てている感覚」を強化
- 常時機能（アンロック対象外）

### 1.3 サイドバー配置

```
1. 取り込む
2. 仕訳する
3. 出力する
4. 突合する  ← 新規追加（事後の検証ステップ）
設定
```

## 2. データソース

| ソース | 入力形式 | 用途 |
|---|---|---|
| 銀行明細 | CSV | 突合の軸（通帳） |
| カード明細 | CSV | 通帳と領収書の橋渡し |
| 請求書（支払い/売上） | PDF / CSV / 写真 | 売掛・買掛の照合 |
| 領収書 | 既存データ（shiwake_records） | 照合先 |

### 2.1 CSVフォーマット対応

- 主要銀行・カード会社のフォーマットを**プリインストール**（開発側で事前登録）
- ユーザーはドロップダウンで銀行/カードを選ぶだけ
- 未対応の銀行/カードはAI自動判定でフォールバック

### 2.2 アップロード解析パイプライン

```
写真 → Haiku（書類種別判定: 通帳？カード明細？領収書？請求書？）→ Sonnet（中身解析）
PDF  → Sonnet（直接解析）
CSV  → フォーマット定義に基づきパース
```

## 3. UI設計

### 3.1 画面構成

- **上部**: 銀行/カード選択ドロップダウン + 月選択 + CSV取り込みボタン
- **サマリーカード**: 明細件数 / 照合済み / 要確認 / 照合率
- **メインテーブル**: 通帳を左軸、各行に照合先を表示
- **フッター**: 通帳残高 vs 帳簿残高 → 差額

### 3.2 テーブル列構成

| 日付 | 通帳の摘要 | 金額 | 状態 | 照合先の証憑 |

### 3.3 状態バッジ

| バッジ | 意味 | 色 |
|---|---|---|
| 一致 | 照合完了 | 緑 |
| 重複疑い | カード明細と領収書の両方あり等 | 赤 |
| 未照合 | 対応する証憑なし | 黄 |
| 欠落 | 定期支払いが今月ない | 紫 |

### 3.4 ゴール

**要確認 0件・照合率 100%** = 突合完了。全数字が合って帳簿を閉じられる状態。

## 4. マッチングロジック

### 4.1 3段階マッチング

| レベル | 条件 | アクション |
|---|---|---|
| Lv1 自動照合 | 金額完全一致 + 日付±3日 + 取引先エイリアス一致 | 自動で「一致」処理 |
| Lv2 候補提示 | 金額完全一致 + 日付±7日 + 名前不一致 | 「この証憑では？」と候補表示 |
| Lv3 未照合 | 合うものなし | 「対応する証憑なし」表示 |

### 4.2 取引先名の照合方法

- 通帳の摘要はカタカナ・短縮表記（例: 「VISAゴディバ」「ﾌﾘｺﾐ ｶ)ﾅﾆﾜｼﾖｳｼﾞ」）
- **取引先マスタにエイリアスとして登録**して解決
- 初回: Lv2候補提示 → ユーザー確認 → エイリアス自動登録
- 次回以降: エイリアスでLv1自動照合
- **使うほど精度が上がる = 育てる設計思想**

### 4.3 1:N マッチング（カード引落）

カード明細が通帳と領収書の「橋渡し」になる3層構造:

```
通帳の1行（VISA引落 ¥52,300）
  ↕ 合計額で照合
カード明細（5件、合計 ¥52,300）
  ↕ 個別に照合
領収書（5枚）
```

- カード明細アップロード時に合計額を自動計算
- 通帳の「VISA引落」行と合計額でマッチ
- カード明細の各行と領収書を個別にマッチ
- 全件照合済み = 「この引落は照合完了」、1件でも欠ければ警告

### 4.4 N:1 マッチング（まとめ払い）

複数請求書を1回の振込で支払うパターン:

```
請求書A ¥50,000 + 請求書B ¥30,000 + 請求書C ¥20,000
  ↕ 組み合わせ合計で照合
通帳の1行（フリコミ ¥100,000）
```

- 1:1で合う請求書がない場合、同一取引先の未払い請求書を検索
- 組み合わせの合計が振込額と一致するか計算
- 実務上、1取引先あたり未払い10件以下 → 全組み合わせ試行でも軽量

### 4.5 請求書マッチングの日付許容

- 請求書 → 支払いのタイムラグがあるため、日付許容は**±60日**
- 1:1/N:1 とも同じ許容幅

### 4.6 差額の扱い

- 振込手数料・値引き・相殺で**金額がぴったり合わない**ケースあり
- → 差額を表示し、ユーザーが理由を選んで解決（振込手数料 / 値引き / その他）
- 解決後「一致」に変わり、照合率が上がる

## 5. 異常検知（突合内で検出するもの）

| 検知タイプ | 内容 |
|---|---|
| 重複検知 | カード明細と領収書の両方が仕訳に存在 → 二重計上アラート |
| 欠落検知 | 毎月ある定期支払い（通信費等）が今月ない → 計上漏れアラート |
| 未払い検知 | 請求書はあるのに通帳に支払いがない → 支払い漏れアラート |
| 不明出金 | 通帳に出金があるのに対応する証憑がない → 証憑不足アラート |


### 6.2 v3.0.0「真のAgent化」の廃止

- 元の定義（会計ソフトAPI連携 + 能動的質問）は現実的に困難
- API連携: freee以外は未公開/要パートナー認定 → CSVに着地済み
- 能動的質問: Webアプリ内では実質不可能 → Elite の対話モード(Slack/LINE)が担う
- v3.0.0 は廃止し、v3.x = Elite に統合

### 6.3 プレゼン戦略

- **兄（税理士）へのプレゼンゴール**: v2.10.0 まで
- **ストーリー**: 「スタッフの作業は変わらない。AIが裏で精度とスピードを上げる」
- **Elite はビジョンとして提示**: 行動変容の壁があるため、実績を積んでから提案
- **スタッフの性質**: 新しいことを嫌う → 導入ハードルの低さが最重要
