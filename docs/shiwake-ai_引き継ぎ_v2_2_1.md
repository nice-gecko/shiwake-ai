### 1.4 ユーザー対話ルール(継承)

#### トークン節約ポリシー
- **コードを書く前に承認を得る**
- 大規模な変更は事前に説明・効果見積もりを提示
- 実装は Claude Code に分担、Web版Claudeは指示出し・判断役

#### ミス防止
- セルフチェック5回(Claude内部での確認)

#### コミュニケーション
- 文章は簡潔に(長文回避)
- DLからの移動コマンド不要、cd後すぐ git add/commit/push の一括コマンドを提示

#### バージョンアップの判断
- 安全な変更は即実装
- リスクがある変更は別バージョン分割を提案
- push時は index.html のバージョン表記もバンプ(セマンティックバージョニング)
  - バグ修正: パッチ+1
  - 機能追加: マイナー+1
  - 破壊的変更: メジャー+1

### 1.5 デプロイコマンド

ローカルファイルは `~/APP/shiwake-ai` 直接配置前提:

```bash
cd ~/APP/shiwake-ai && \
node -c server.js && \
git add [変更ファイル] && \
git commit -m "vX.Y.Z: 変更内容" && \
git push origin main
```

GitHubへのpushでRenderが自動デプロイ。

### 5.4 重要な戦略判断:法務監修のタイミング

Gemini が提示し、運営者として採用した戦略:

**「数社の実戦投入を経てから弁護士監修」**

理由:
- 今すぐ弁護士監修(数万〜十数万円) → 実戦経験なしの机上監修
- 数社実戦投入後に弁護士監修 → 実務課題を反映した監修

→ コスパが良く、実態に即した監修が可能

### 5.5 運用上の重要ルール(v1.4 第14条との整合維持)

第14条の業法ガードを規約だけのものにしないため、以下を維持:

1. **本サービスのUI上で「仕訳の確定処理機能」を追加しない**(CSV出力後は会計ソフトに委ねる設計を維持)
2. **営業資料・サポート資料**で本サービスを「確定ツール」ではなく「**候補生成・CSV出力ツール**」と一貫して説明
3. **将来UIで確定処理機能を追加する場合**は、第14条の整理に影響するため、規約改定および税理士法上の再検討を要する

## 7. 設計思想(継承事項)

### 7.1 ユーザー体験の核
**「何をどう判断させるかを明確に見せること」が継続利用につながる**

- ユーザーが自分専用ツールを育てている感覚を大事にする
- 専門性の強化感を持たせる
- 育成度・最頻マスタ・マスタ学習アニメーション・取引先マスタはこの象徴

### 7.2 機能追加時の指針
- 機能追加時は常に「ユーザーが自分のツールとして育てている実感」を意識する
- 透明性のあるフィードバック(マスタヒット率・キャッシュヒット率)を提供
- ルール学習UIはこの思想の象徴

### 7.3 AI仕訳精度ファースト
- コスト最適化より仕訳精度を優先
- マスタ学習データ汚染懸念があれば実装しない

### 7.4 セルフ差別化(高価格モデル成立)
廉価版(AI SaaS版)があるから、高価格版(Agent版)の特別感が伝わる。
**比較対象がないと、高級品は売れない**(マーケの定石)。

### 7.5 メタファーは2つを併用する
- **コピー** = 人間メタファー(ルーキー、ジュニア、シニア、エージェント等のキャリアパス)
- **機能名** = 能力メタファー(Agent認定、自動取り込み、自動承認等)

### 7.6 「アンロック」と「常時機能」を区別する
- **アンロック**: ユーザーが信頼を委ねる代わりに自動化される
- **常時機能**: プロダクトの安全装置(自動化が進むほど必要)
- 異常検知アラートは常時機能。「特典」として後から解放してはいけない

### 7.7 プッシュ→プルの転換
月次レポートを「勝手に届く」ではなく「**ダッシュボードで見に行く**」設計に。
ユーザー主体性を尊重するUXは、Agent的発想と整合する。

### 7.8 会計ソフトAPI連携の現実解(2026年5月時点)
- freee: API成熟(連携可能)
- マネーフォワード: 仕訳API未公開
- 弥生・勘定奉行: クローズドAPI(パートナー認定が必要)

→ 「自動エクスポート」は**APIではなくCSV+フォルダ整理に着地**。全会計ソフト対応可能、現実解として強い。

### 7.9 業法ガード設計(v2.2.1で確立)
shiwake-ai は「**CSV出力する計算補助ツール**」であり、確定行為は会計ソフトで人間が行う。
これは規約上の表現にとどまらず、プロダクト実態として維持する。

→ 将来的にUIで確定処理機能を追加する場合は、税理士法上の再検討が必要。
