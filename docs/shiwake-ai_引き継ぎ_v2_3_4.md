## 🔍 補足: Render Cron について(2026-05-12 結論)

戦略メモ(2026-05-10)で「Render Cron 月次バッチ有料化」が積み残し項目に上がっていたが、本日改めて検討した結果:

**現状の月次処理はすべてリアルタイム計算で実装されている**
- 月次カウントリセット: ユーザー画面アクセス時に billing_period_end をチェックして自動リセット(server.js:1325 付近)
- 代理店ランク判定: 代理店ダッシュボード表示時に monthlyVolume を再計算(server.js:1438 付近)

つまり「決まった時刻に自動で動かすバッチ」は現時点で不要。

**Render の有料化($7/月)も現時点では先送り**
- Cron Job 機能は Render 有料プランから利用可能
- 必要になるのは「ユーザーが画面を開かなくても月初に強制リセット」のようなケース
- 代理店契約者0人 + ユーザーアクセスもまだ少ない現状では、リアルタイム計算で十分

**判断**: 本格営業に出る直前(または契約代理店が10件超え)のタイミングで再判断する。それまではスキップ。

### スマホ撮影モード構想(新規)

#### 着想
自動取り込みを実運用する現場では、人間は「とにかく写真を撮る役」になる。
現ルート: スマホで「撮影→解析→仕訳」一気通貫
追加ルート案: スマホで「撮影→自動切り取り→Dropbox蓄積」、処理は後で PC でまとめて

#### 採否
**採用。ただし v2.4.0 完走後に投入**。

#### 採用理由
- 経理現場では「撮るタスク(出先・隙間時間)」と「処理タスク(PC前で集中)」が時間的に分離している
- 既存の Dropbox 自動取り込みパスを再利用できる(出口側はほぼ既存資産)
- PC版はスキャン機能非表示で良い(PC で写真を撮る人はいない)

#### 実装メモ
- 書類検出ライブラリ: jscanify か OpenCV.js を検証
- スマホブラウザでの精度・速度は実機検証必須
- 位置づけは**判断系ではなく利便系**。信頼度連動・プラン特典には組み込まない(設計思想を混ぜない)
- PC/スマホで UI を出し分け(モバイル時のみ撮影モードを前面に出す)

#### 兄に見せる事前デモとして優先順を入れ替える選択肢もある
撮影体験のインパクトが強いので、未振り分けトレイより先に出す判断もあり得る。投入前に再確認。

### 楽楽精算との比較(競合分析)

#### 結論: 別領域。住み分け一択

#### 根本的な違い

| | 楽楽精算 | shiwake-ai |
|---|---|---|
| 本質 | 経費精算ワークフロー | 仕訳生成エンジン |
| 業務範囲 | 申請→承認→精算→振込→仕訳CSV | 証憑→仕訳CSV |
| 対象証憑 | 社員立替の領収書のみ | 領収書・請求書・通帳明細など全証憑 |
| 仕訳のしくみ | 申請項目に勘定科目を事前マスタで紐付け(ルールベース) | Claude が画像を直接読んで仕訳生成+取引先学習(AIベース) |
| 必要な前提 | 経費精算ワークフローが業務に存在すること | 証憑があること(それだけ) |
| ターゲット | 社員数100名以上の中堅〜大企業 経理部 | 税理士事務所/個人事業主/中小法人 |
| 価格帯 | 月3万円〜(初期費10万円〜) | 月980円〜 |
| 規模 | 累計2万社導入の市場リーダー | 未ローンチ |
| 会計ソフト連携 | CSV出力(直接APIではない) | CSV出力(同じ土俵) |

#### 1行で説明する場合
「楽楽精算は**社員の経費精算ワークフロー**が主役。shiwake-ai は**証憑を AI が読んで仕訳を作る**のが主役。重なるのは『領収書をスマホで撮る』だけで、その前後の業務が全然違う」

#### ターゲット戦略
- **楽楽精算ユーザーを奪う動きはしない**: 機能ゲームで勝てない(20年・2万社の蓄積、ワークフロー機能群が完成済み)
- **狙う層**: 経費精算ワークフローが不要 or 簡素だが、仕訳作業は発生する層
  - 個人事業主・フリーランス
  - 社員5〜30名の小規模法人
  - **税理士の顧問先**(現行B2B戦略と一致)
  - 売上請求書・仕入請求書・通帳明細まで含めた全証憑の仕訳が必要な層
- **将来的な併用余地**: 楽楽精算ユーザーでも、経費精算外の証憑(売上請求書等)の仕訳で shiwake-ai を使う余地あり。API連携は遠い将来の話

### カウントアップ追従バッジ構想(B案で確定)

#### 背景
仕訳数・取引先マスタ数のカウントアップアニメは、設計思想「ユーザーが自分専用ツールを育てている感覚」の核。
ただし現状は画面上部にあり、実作業(仕訳カード承認)は画面下部で行うため、ユーザーが気付かない問題がある。

#### 採用案: B案(スクロール追従バッジ)
- スクロールするとメイン上部の2カード(スタッフインセンティブ/取引先マスタ)は通常通り画面外へ
- 代わりに**画面右上に小さなピル型バッジ**がふわっと出現
- バッジ内容: `[アイコン] 累計N枚 · [アイコン] 学習M社`
- カウントアップ時はバッジが脈動(scale+ring expand のアニメーション)
- スクロールが上に戻ったらバッジはふわっと消える

#### A案(1行sticky)を不採用にした理由
- メインカードの情報と sticky 行の情報がダブる(累計枚数・学習取引先・残承認数を両方に出す形)
- 情報量が増えてかえって認知負荷が上がる
- 「気持ちよさ」より「情報密度」が前に出てしまう

#### 実装メモ
- 位置: 画面右上(バグ報告FABは右下なので衝突しない)
